⏺ Good — clean, readable code. Here's what I'd prioritize, roughly in order:

  ---
  1. localStorage persistence (highest impact, low effort)
  Store tasks as a JSON array in localStorage. On page load, read and re-render
  them. This makes the app actually useful with one change to main.js.

  2. Mark complete (core study tracker behavior)
  Add a checkbox or click-to-complete on each task. Strike through completed
  tasks and move them to a "Done" section. Pairs naturally with persistence.

  3. Sort by due date (improves usability immediately)
  Right now tasks appear in insertion order. Auto-sorting by due date (soonest
  first) makes the list much more actionable without adding UI complexity.

  4. Priority field (your feature card already promises it)
  Add a simple High / Medium / Low select to the form. Show a colored dot per
  priority. Pairs with sorting — sort by due date, then by priority.

  5. Filter/group by class (useful once list grows)
  A simple dropdown filter ("All classes" or a specific class name) lets
  students focus. You can derive the class list from the task data — no extra
  form needed.

  ---
  What I'd skip for now:
  - A calendar view — complexity is high, payoff is low until you have many
  tasks
  - User accounts / a backend — massively increases scope; localStorage handles
  90% of the use case
  - Notifications/alerts — requires service workers or a backend, and browser
  APIs for this are cumbersome

  ---
  Suggested next step: Start with localStorage + mark-complete together —
  they're tightly coupled (you need to persist the completed state anyway), and
  together they make the app feel real. Want me to implement those?

  Adding a task
  - Fill in all three fields and submit — task appears in the list
  - Form clears and focus returns to the assignment name field
  - Submit with any field empty — nothing happens (browser validation fires)

  localStorage persistence
  - Add a task, then open DevTools → Application → Local Storage → check
  studyflow-tasks contains the task as JSON
  - Refresh the page — task still appears

  Mark complete / incomplete
  - Check the checkbox — name gets strikethrough, due date color goes grey
  - Refresh — task re-renders with the checkbox still checked and strikethrough
  applied
  - Uncheck — strikethrough clears, due date color returns
  - Refresh again — unchecked state persists

  Deleting a task
  - Click × — task disappears from the list and from localStorage
  - Refresh — deleted task does not come back

  Empty state
  - Delete all tasks — "No assignments yet" message reappears
  - Refresh with no tasks in storage — empty state shows on load (not a blank
  list)

  ---
  Edge cases to watch for

  - Completed task color override — if a task is both completed and overdue, the
   inline style color on .tl-meta will win over the CSS rule .completed 
  .tl-meta. You'd see red instead of grey. The !important in the CSS handles
  this, but worth verifying visually.
  - Date.now() as ID — IDs are millisecond timestamps, so adding two tasks in
  the same millisecond (unlikely by hand, possible in tests) would produce a
  duplicate ID. Not a real-world issue for a demo, but something to know.
  - Corrupted localStorage — if studyflow-tasks contains invalid JSON (e.g.
  manually edited in DevTools), JSON.parse will throw and the page will break.
  You can test this by setting the key to "bad" in DevTools and refreshing.

⏺ Found it. The script tag in index.html is type="module":

  <script type="module" src="main.js"></script>

  ES modules cannot load over file:// — browsers block them due to CORS
  restrictions. If you're opening index.html by double-clicking it (or dragging
  it into the browser), the module silently fails to load, so no event listeners
   are ever attached, and the form falls back to its default behavior: a
  page-jump submit.

  The fix is just to use the Vite dev server, which is already set up:

  npm run dev

  Then open the localhost URL it prints (typically http://localhost:5173).

  ---
  How to confirm this is the cause:
  1. Check the browser console when opening via file:// — you'll likely see a
  CORS/module error
  2. Run npm run dev, open the localhost URL, and the form should work
  immediately

  Why it was working before the changes: it wasn't — type="module" was already
  there from the initial setup. The previous version also would have failed the
  same way. You may just not have tested the form via file:// before.
